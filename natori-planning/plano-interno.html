Visão Geral do Produto

Uma plataforma web privada que centraliza o monitoramento competitivo da Natori Foods nos três principais marketplaces brasileiros: Mercado Livre, Shopee e Amazon. O sistema coleta preços, rastreia Buy Box, analisa reviews, e entrega diagnósticos via IA para que a equipe Natori tome decisões rápidas e fundamentadas — sem depender de planilhas ou verificação manual.

O público é interno: o próprio Kaio e a equipe de operações. Não é um produto SaaS. A premissa é simples: 55 produtos Curva A representam R$815K/ano. Perder Buy Box em qualquer deles por 48h custa dinheiro real. A plataforma transforma esse risco em visibilidade e, eventualmente, em automação de resposta.

A proposta de valor em uma linha: menos tempo investigando, mais tempo decidindo. A plataforma agrega dados que hoje são coletados à mão ou nem são coletados, e usa IA para transformá-los em ação.

MÉDIO

2

Mercado Livre Operacional

Primeiro valor financeiro mensurável. ROI começa aqui.
O que entrega
  • Coleta automatizada dos 55 SKUs Curva A
  • Monitoramento de preços de concorrentes em tempo real
  • Detecção de perda de Buy Box com alertas no painel
  • Cadastro editável de produtos e concorrentes
  • Histórico de preços começando a acumular
  • Dashboard ML com filtros e busca
Componentes técnicos
  • Worker ML: API oficial Mercado Livre + scraper de concorrentes
  • Tabelas: skus, competitors, price_history, buybox_events
  • Cron job a cada 30 minutos para 55 SKUs
  • Páginas: /dashboard, /buybox, /catalogo
  • Sistema de alertas internos (painel + email)
Esforço estimado
ALTO
Dependências

Fase 1 completa. Token API do Mercado Livre da Natori (requer conta developer ML ativa). Lista final dos 55 SKUs confirmada pelo Kaio.

Riscos específicos
Rate limit ML: API oficial limita 60 req/min por token. Com 55 SKUs + concorrentes, planejar batching e cache inteligente.
API inconsistente: ML não expõe todos os dados de concorrente via API oficial. Para dados de terceiros, scraping da página do produto pode ser necessário — frágil por natureza.
Conta developer: Kaio pode não ter conta developer ML configurada. Criar isso no kickoff, não no início da fase.
Definição de "pronto"
Pronto quando: todos os 55 SKUs aparecem no dashboard com preço e status de Buy Box atualizados em menos de 30 minutos, sem intervenção manual.
3

Cobertura Completa dos Marketplaces

Os três canais unificados. Amazon é o maior risco técnico do projeto.
O que entrega
  • Integração com Shopee (API parcial Brasil)
  • Coleta de dados Amazon via scraping
  • Dashboard unificado das 3 plataformas
  • Comparativo cruzado de preços e Buy Box
  • Alertas consolidados por produto
Componentes técnicos
  • Worker Shopee: Shopee Open Platform API (Brasil)
  • Worker Amazon: Playwright/Puppeteer + proxy residencial
  • Unificador de dados: normalização de IDs entre plataformas
  • Adapter layer: cada marketplace tem schema diferente
  • Dashboard refatorado para 3 fontes
Esforço estimado
MUITO ALTO
Dependências

Fase 2 estável e com pelo menos 2 semanas de dados acumulados. Credenciais Shopee do cliente. Decisão sobre proxy para Amazon.

Riscos específicos
Amazon — RISCO PRINCIPAL DO PROJETO: Amazon ativamente combate scrapers. Anti-bot agressivo, CAPTCHAs, bloqueios de IP. Proxy residencial rotacionado provavelmente necessário (+custo ~R$200/mês). Pode quebrar sem aviso após update do Amazon. Tratar Amazon como "best effort" no contrato: dados disponíveis quando possível, mas sem SLA de atualização.
Shopee API Brasil: Documentação escassa, mudanças sem aviso prévio, suporte lento. Endpoints para dados de concorrentes são limitados. Pode ser necessário scraping complementar.
Normalização de SKUs: O mesmo produto tem IDs diferentes em ML, Shopee e Amazon. Mapeamento manual inicial pode ser necessário para os 55 SKUs.
Definição de "pronto"
Pronto quando: os 3 marketplaces aparecem no dashboard lado a lado com dados coerentes, e o Kaio consegue ver o Buy Box status de um produto nas 3 plataformas em uma tela.
4

IA Ativa no Dia a Dia

A plataforma começa a pensar. Requer histórico mínimo de 2-3 semanas.
O que entrega
  • Diagnóstico IA para quedas de vendas e anomalias
  • Simulador de impacto de preço (modelo estatístico)
  • Análise automática de reviews dos clientes
  • Previsão de ruptura de estoque
  • Detecção de padrões e sazonalidades
Componentes técnicos
  • Claude Haiku: diagnóstico diário de anomalias (barato)
  • Claude Sonnet: análises profundas on-demand
  • Pipeline z-score para detecção de anomalias estatísticas
  • Modelo de elasticidade simples para simulador de preço
  • Embeddings + clustering para agrupar e classificar reviews
Esforço estimado
MÉDIO
Dependências

Fases 2 e 3 com histórico mínimo de 2-3 semanas acumulado. Sem histórico, os modelos estatísticos são pouco confiáveis e a IA vai alucinar por falta de contexto real.

Riscos específicos
Alucinação da IA: Se o prompt não tiver contexto suficiente (histórico, produto, categoria), Claude pode dar diagnósticos plausíveis mas incorretos. Estruturar prompts com dados concretos, não texto genérico.
Custo API crescendo: Se cada usuário fizer 50 diagnósticos/dia, o custo escala. Hard limits por conta, cache agressivo de respostas similares, usar Haiku por padrão.
Modelo de elasticidade impreciso: Com menos de 3 meses de dados, a correlação preço-volume é estatisticamente fraca. Comunicar isso como "estimativa" no UI, não como previsão.
Definição de "pronto"
Pronto quando: Kaio faz 5 perguntas ao diagnóstico sobre produtos reais e as 5 respostas são corretas, acionáveis e baseadas em dados do sistema.
5

A Plataforma Agindo Sozinha

Automação de alto risco. Regras de preço exigem múltiplas salvaguardas obrigatórias.
O que entrega
  • Engine de regras automáticas if/then para ajuste de preço
  • Alertas WhatsApp reais (não simulados) para eventos críticos
  • Relatórios executivos semanais gerados automaticamente
  • Respostas sugeridas a reviews (requer aprovação humana)
  • Sistema de notificações multicanal configurável
Componentes técnicos
  • Engine de regras: declarativo, auditável, dry-run mode
  • Limites mínimos obrigatórios: preço nunca abaixo de X
  • Kill switch global para parar toda automação
  • Evolution API (self-hosted) ou Twilio para WhatsApp
  • PDF generator (Playwright) para relatórios
  • Cron scheduler com retry e alertas de falha
Esforço estimado
ALTO
Dependências

Fases 1 a 4 estáveis. A automação pressupõe dados confiáveis — regras automáticas sobre dados ruins causam dano real. Número WhatsApp Business aprovado.

Riscos específicos
ALTO RISCO — Regra baixando preço abaixo do custo: Bug em lógica condicional pode fazer o sistema baixar preço para zero ou abaixo do custo. Obrigatório: limite mínimo configurado pelo cliente, log de auditoria de cada ajuste, aprovação humana para os primeiros 30 ajustes, kill switch global de 1 clique.
WhatsApp Business API: Custo por mensagem enviada (R$0,05 a R$0,10). Templates precisam de aprovação da Meta — processo leva dias. Não prometer entrega de WhatsApp no mesmo sprint de configuração.
Falhas silenciosas no scheduler: Se o cron falha e não alerta, a Natori fica sem dados sem saber. Monitoramento de health check obrigatório.
Definição de "pronto"
Pronto quando: Kaio configura 1 regra, ela dispara corretamente em ambiente real, e ele recebe o alerta WhatsApp correspondente no número dele.

Escala e Inteligência Coletiva

Do operacional ao estratégico. Maturidade da plataforma.
O que entrega

  • RBAC completo com permissões granulares por função
  • Fichas completas de cada concorrente rastreado
  • Importação em massa via planilha Excel/CSV

Componentes técnicos

  • Competitor profiles: histórico, variações de preço, market share estimado

Esforço estimado
MÉDIO-ALTO
Dependências

Todas as fases anteriores. Saber qual ERP a Natori usa (Bling? Omie? TOTVS?) para planejar a integração. RBAC só faz sentido se a equipe tiver 3+ usuários ativos.

Riscos específicos

Adoção da equipe: Dashboard de performance da equipe só entrega valor se a equipe realmente usar a plataforma diariamente. O risco não é técnico — é de adoção. Onboarding assistido é obrigatório.
Definição de "pronto"
Pronto quando: 3 ou mais usuários distintos da Natori usam a plataforma com papéis e permissões diferentes, sem necessidade de suporte no uso básico.

Engenharia

Desafios Técnicos Principais

Coleta confiável na Amazon
Amazon investe ativamente em detectar e bloquear scrapers. O desafio não é técnico (ferramentas existem), é operacional: manter a coleta funcionando depois de bloqueios, CAPTCHAs e mudanças de layout. Playwright com fingerprinting evasion (playwright-extra + stealth plugin) é o ponto de partida. Se a taxa de bloqueio for alta, proxies residenciais rotativos são necessários — custo adicional de ~R$200/mês.

A estratégia de mitigação mais sólida é contratual: Amazon no escopo como "best effort". A Natori recebe dados da Amazon quando disponíveis, mas o SLA de atualização só se aplica a ML e Shopee. Isso precisa estar no contrato antes de começar.
Série temporal eficiente com TimescaleDB
Price history vai crescer rápido: 55 SKUs x 3 marketplaces x 48 checks/dia = ~7.900 registros/dia só de preços, sem contar concorrentes. Em 6 meses, são ~1,4M de registros só no modelo base.

Política de downsample obrigatória: manter granularidade de 15 minutos por 30 dias, 1 hora por 6 meses, 1 dia por 2 anos. TimescaleDB facilita isso com continuous aggregates e retention policies automáticas. Configurar desde a Fase 1 evita migração traumática depois.
Prompts de IA com contexto correto
O diagnóstico da IA é tão bom quanto os dados que recebe. Prompt mal estruturado = resposta genérica ou alucinação. A abordagem correta é injetar dados específicos: produto, histórico de 30 dias, eventos de Buy Box, variação de preço dos concorrentes, sazonalidade identificada.

Usar Claude Haiku para diagnósticos diários automáticos (barato, ~$0.25/M tokens). Reservar Sonnet para análises profundas pedidas explicitamente pelo usuário. Cache de respostas para perguntas similares sobre o mesmo SKU no mesmo dia. Hard limit de tokens por conta para evitar custo não controlado.
Sincronização de Buy Box em tempo real
Buy Box pode mudar várias vezes ao dia, especialmente em produtos competitivos. O trade-off é frequência de check vs. custo de API vs. tempo de detecção. Para os 55 SKUs Curva A, checar a cada 30 minutos é o balanço razoável para fase inicial.

Para produtos de alto giro onde perder Buy Box por 2 horas tem impacto financeiro significativo, implementar webhook ou polling mais agressivo como feature premium futura. ML tem webhooks de notificação de Buy Box que podem eliminar o polling por completo quando ativados corretamente.
Regras automáticas seguras
Uma regra que ajusta preço automaticamente é o feature mais arriscado do projeto. Um bug de lógica pode causar prejuízo financeiro direto e imediato. A implementação precisa de múltiplas camadas de proteção obrigatórias: preço mínimo configurado pelo cliente (nunca ultrapassável), dry-run mode para testar regras sem executar, log imutável de cada ajuste com razão, kill switch global que para tudo em 1 clique, e aprovação humana obrigatória para as primeiras N execuções de cada regra nova.

Nunca lançar esta feature sem o Kaio ter lido e aprovado explicitamente as salvaguardas em reunião. Documentar no contrato que ajustes automáticos têm limites e que o cliente é responsável pela configuração dos preços mínimos.
Arquitetura multi-tenant desde o inicio
Se a plataforma der certo com a Natori, a pergunta natural em 6 meses é: "posso vender isso para outra importadora?" A resposta depende de como o banco foi modelado na Fase 1. Schema por tenant (cada cliente tem schema separado no Postgres) ou Row-Level Security com tenant_id em todas as tabelas?

RLS com tenant_id é mais simples de manter e escala bem para dezenas de clientes. Schema por tenant dá mais isolamento mas é mais difícil de migrar e operar. Recomendação: RLS desde a Fase 1, mesmo que só haja 1 tenant. O custo de mudar depois é alto.

  1. 1

    Fase 1 — Fundação

    Bloqueia tudo. Sem infra e banco corretos, nada mais funciona. Prioridade máxima, sem negociação.

  2. 2

    Fase 2 — Mercado Livre

    Primeiro valor financeiro real. ML é o maior canal da Natori. ROI imediato e mensurável. Justify o R$8K/mês nesta fase.

  3. 3

    Fase 4 parcial — Diagnóstico IA no ML

    Adiantar o diagnóstico IA já durante a Fase 2, assim que houver 2 semanas de dados. É de esforço médio, encanta o cliente, e diferencia a plataforma de um "simples monitorador de preços".

  4. 4

    Fase 3 — Shopee + Amazon

    Expandir cobertura após ML estável. Amazon com risco gerenciado e declarado no contrato.

  5. 5

    Fase 5 — Automação

    Somente após dados confiáveis das 3 plataformas. Automação sobre dados ruins é perigoso.

  6. 6

    Fase 6 — Escala

    Plataforma madura. Adicionado conforme necessidade real da equipe — não por antecipação.

Nota sobre paralelismo: Fases nao sao estritamente sequenciais. Fase 4 (diagnóstico IA) pode adiantar assim que Fase 2 tiver 2 semanas de dados. Fase 5 (WhatsApp) pode começar a configurar templates enquanto Fase 3 ainda está em andamento.

Aberto

Decisões Pendentes

Questões que precisam ser decididas antes ou durante o projeto. Cada uma tem impacto no custo, prazo ou escopo.

Stack definitiva: Astro + React vs. Next.js. Astro é mais leve e já é stack conhecida. Next.js tem server actions e melhor suporte a SSR complexo. Para este projeto, Astro é suficiente se não houver necessidade de SSR pesado.

Supabase vs. Postgres self-hosted: Supabase tem Auth, Storage e API REST embutidos — acelera a Fase 1. Postgres self-hosted em Railway ou Fly.io custa menos na escala. Para fase inicial, Supabase é a escolha racional.

WhatsApp: Evolution API vs. Twilio: Evolution API self-hosted é gratuito (além do hosting) mas mais frágil e fora do oficial. Twilio é estável, aprovado pela Meta, mas tem custo por mensagem. Para um contrato de R$8K/mês, Twilio é o mais seguro.

Precificação de features extras: O que acontece quando o Kaio pede algo fora do escopo das 6 fases? Definir um valor por hora de change order antes do contrato ser assinado. Evitar definir isso no calor do momento.

Exclusividade de segmento: O contrato prevê exclusividade — você nao faz esta plataforma para outra importadora de alimentos asiáticos? Se sim, precisa estar no contrato e idealmente ter valor adicional. Se nao, deixar claro que pode replicar o modelo para outros clientes.

SLA de bugs em produção: Qual é o tempo de resposta garantido para bug crítico (coleta parada) vs. bug menor (UI)? Definir no contrato: ex. crítico em 4h, alto em 24h, médio em 72h. Impacta diretamente a percepção de qualidade do serviço.
