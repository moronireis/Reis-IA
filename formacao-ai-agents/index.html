Entendendo como modelos de linguagem funcionam e como usa-los

O que e um LLM

Um Large Language Model (LLM) e uma rede neural treinada em bilhoes de textos — livros, artigos, codigo, paginas web — com o objetivo de aprender a prever qual token vem a seguir em uma sequencia. A ideia central e surpreendentemente simples: dado o contexto anterior, qual e a palavra mais provavel? Repetido bilhoes de vezes durante o treinamento, esse processo gera um modelo capaz de raciocinar, escrever, traduzir e programar.

O que diferencia os LLMs modernos das gerações anteriores e a escala. GPT-3 tinha 175 bilhoes de parametros. Claude 3 Opus tem estimativa similar. Esses parametros sao os "pesos" da rede neural — numeros ajustados durante o treinamento que codificam o conhecimento do modelo sobre linguagem, fatos e raciocinio.

Tokenizacao — voce paga por token, nao por palavra

LLMs nao leem palavras. Eles leem tokens — fragmentos de texto que podem ser palavras inteiras, partes de palavras, ou ate caracteres individuais. Em ingles, a regra pratica e ~4 caracteres por token. Em portugues, por causa das palavras longas e acentos, o ratio e um pouco pior: ~3-3.5 caracteres por token.

Por que isso importa? Porque todos os providers cobram por token (input + output). Saber contar tokens te ajuda a estimar custo, otimizar prompts e entender limites de contexto. GPT-4o tem janela de 128K tokens. Claude tem ate 1 milhao de tokens de contexto.

# Contando tokens com tiktoken (Python)
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")
text = "Construindo sistemas multi-agentes com Claude Code"
tokens = enc.encode(text)
print(f"Texto: {text}")
print(f"Tokens: {len(tokens)}")  # ~10 tokens
print(f"IDs: {tokens}")

Embeddings — significado como coordenadas matematicas

Antes de processar tokens, o modelo os converte em embeddings: vetores de centenas ou milhares de numeros que representam o significado do token no espaco matematico. Tokens semanticamente similares ficam proximos nesse espaco. "Rei" e "Rainha" ficam proximos. "Cachorro" e "Gato" ficam proximos. "Python" (linguagem) e "Python" (cobra) ficam em regioes diferentes, dependendo do contexto.

Essa representacao e a base de toda a compreensao do modelo. Voce pode usar embeddings diretamente para busca semantica, clustering e sistemas de recomendacao — mesmo sem gerar texto.

Mecanismo de Atencao (Transformer)

O Transformer — arquitetura publicada pelo Google em 2017 no paper "Attention Is All You Need" — resolveu o problema central de processar sequencias longas. O mecanismo de atencao (self-attention) permite que cada token "olhe" para todos os outros tokens na janela de contexto e calcule o quanto cada um e relevante para interpretar o token atual.

Na pratica: quando o modelo le "O banco onde eu sentei estava molhado", o mecanismo de atencao conecta "banco" com "sentei" e "molhado" — inferindo que e um banco de praca, nao uma instituicao financeira. Esse contexto e calculado em paralelo para todos os tokens, tornando o processo muito mais eficiente que as RNNs antigas que processavam token por token.

O que e Prompt Engineering

Prompt engineering e a pratica de formular instrucoes para LLMs de forma que maximizem a qualidade, precisao e utilidade da resposta. Nao e "hackear" o modelo — e aprender a se comunicar efetivamente com um sistema que responde a linguagem natural com sensibilidade a contexto, estrutura e exemplos.

A diferenca entre um prompt ruim e um bom pode ser a diferenca entre uma resposta generica inutil e uma saida pronta para producao. E uma habilidade que se paga em qualidade de output, reducao de iteracoes e custo menor (prompts mais eficientes geram respostas mais diretas).

Zero-shot, Few-shot e Chain-of-Thought

Zero-shot e simplesmente fazer a pergunta sem fornecer exemplos. Funciona bem para tarefas simples e modelos grandes. Few-shot inclui 2-5 exemplos do formato esperado antes da pergunta real — particularmente eficaz para classificacao, extracao e formatacao especifica. Chain-of-thought (CoT) instrui o modelo a "pensar passo a passo", o que melhora dramaticamente raciocinio matematico e logico.

# Zero-shot
prompt_zero = "Classifique este email como URGENTE ou NORMAL: 'Servidor caiu, site fora do ar'"

# Few-shot
prompt_few = """Classifique emails como URGENTE ou NORMAL.

Email: "Reuniao amanha as 10h" → NORMAL
Email: "Cliente cancelou contrato hoje" → URGENTE
Email: "Servidor caiu, site fora do ar" → """

# Chain-of-Thought
prompt_cot = """Resolva passo a passo:
Uma empresa tem 240 funcionarios. 30% trabalham remotamente.
Quantos funcionarios estao no escritorio?

Raciocine passo a passo antes de dar a resposta final."""

System Prompts e Roles

O system prompt e executado antes de qualquer mensagem do usuario e define o comportamento base do modelo para toda a conversa. E onde voce define persona, restricoes, formato de saida, contexto do negocio e qualquer regra que deve ser sempre aplicada. Pense como a "constituicao" da conversa.

import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=1000,
    system="""Voce e um analista financeiro senior especializado em SaaS.

REGRAS:
- Sempre responda em portugues brasileiro
- Cite metricas especificas (ARR, churn, LTV, CAC)
- Se faltar dados para uma analise confiavel, diga explicitamente
- Formato: analise concisa, bullet points para acoes recomendadas
- Tom: consultivo, direto, sem jargao desnecessario""",
    messages=[
        {"role": "user", "content": "Nossa churn mensal esta em 5%. O que fazer?"}
    ]
)

Formatacao — XML Tags e Delimitadores

Claude responde particularmente bem a XML tags para separar secoes do prompt. Isso reduz ambiguidade e ajuda o modelo a entender exatamente o que e instrucao, o que e contexto e o que e a tarefa real. Use <context>, <instructions>, <examples>, <output_format>.

prompt = """
<context>
Voce esta analisando feedback de clientes de um SaaS B2B.
Base de clientes: 500 empresas, ticket medio R$2.000/mes.
</context>

<instructions>
Analise o feedback abaixo e extraia:
1. Problema principal (1 frase)
2. Sentimento (positivo/neutro/negativo)
3. Acao recomendada (1 frase)
</instructions>

<feedback>
{feedback_text}
</feedback>

<output_format>
Responda em JSON com as chaves: problema, sentimento, acao
</output_format>
"""

Patterns Avancados

  • ▶ Persona: "Voce e um engenheiro senior do Google com 15 anos de experiencia em sistemas distribuidos."
  • ▶ Constraints: "Responda em no maximo 3 paragrafos. Nao use bullet points. Cite pelo menos 1 exemplo real."
  • ▶ Output format: "Responda sempre em JSON valido com as chaves: titulo, resumo, tags[]."
  • ▶ Self-correction: "Apos escrever sua resposta, releia e corrija qualquer inconsistencia logica antes de finalizar."

Importante

Prompts longos nao sao necessariamente melhores. LLMs sofrem de "lost in the middle" — tendem a ignorar informacao no meio de contextos muito longos. Coloque instrucoes criticas no inicio ou no final do prompt.

O que e uma REST API

Uma REST API e uma interface que permite que seu codigo se comunique com um servico externo via HTTP. Para LLMs, o fluxo e simples: voce faz um POST request com seu prompt no body, a API processa e devolve a resposta em JSON. Todo o processamento pesado acontece nos servidores do provider — voce so envia texto e recebe texto.

A autenticacao e feita via API key — um token secreto que o provider te da quando cria sua conta. Essa chave nunca deve aparecer no seu codigo fonte ou ser commitada no git. A pratica correta e armazena-la em variaveis de ambiente (.env) e acessar com os.environ.get().

Importante — Seguranca

NUNCA coloque API keys diretamente no codigo. Use sempre variaveis de ambiente. Adicione .env ao seu .gitignore. Uma key exposta no GitHub pode gerar centenas de dolares em custos em minutos por bots que escaneiam repositorios publicos.

Streaming — texto chega em tempo real

Por padrao, a API espera o modelo gerar toda a resposta antes de devolver. Com streaming, os tokens chegam conforme sao gerados — assim como voce ve no Claude.ai. Isso melhora drasticamente a experiencia do usuario em aplicacoes, pois elimina a sensacao de "congelamento" em respostas longas.

# Streaming com Claude
import anthropic

client = anthropic.Anthropic()

with client.messages.stream(
    model="claude-3-5-sonnet-20241022",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Escreva um relatorio de 500 palavras sobre AI."}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)  # imprime token por token
print()  # nova linha ao final

Parametros importantes

  • ▶ temperature (0-1): Controla aleatoriedade. 0 = sempre a resposta mais provavel (deterministica). 1 = mais criativo e variado. Para extracao de dados use 0. Para escrita criativa use 0.7-1.
  • ▶ max_tokens: Limite maximo de tokens na resposta. Nao confunda com janela de contexto — isso so limita o output.
  • ▶ top_p (0-1): Alternativa a temperature. Considera apenas os tokens que somam P% de probabilidade cumulativa. top_p=0.9 e uma boa configuracao geral.
  • ▶ stop sequences: Strings que fazem o modelo parar de gerar. Util para garantir que o output termine num formato especifico.

Por que rodar modelos localmente

Modelos locais tem quatro vantagens fundamentais que os tornam indispensaveis em certos cenarios: privacidade (nenhum dado sai da sua maquina — ideal para dados sensiveis de clientes), custo zero por chamada (paga so pela eletricidade), sem rate limits (pode fazer milhares de chamadas em paralelo), e funciona offline (critico para ambientes sem internet ou com latencia alta).

A contrapartida e qualidade. Um Llama 3.3 70B rodando local e excelente, mas nao iguala Claude 3.7 Sonnet para tarefas complexas de raciocinio. Para prototipagem, desenvolvimento e dados sensiveis, local e perfeito. Para producao com qualidade maxima, use cloud.

Ollama — o jeito mais simples de rodar modelos

Ollama e uma ferramenta de linha de comando que gerencia download, quantizacao e execucao de modelos locais. Funciona como um "Docker para LLMs". Apos instalacao, um comando e suficiente para baixar e rodar qualquer modelo suportado.

# Instalacao (Mac/Linux)
curl -fsSL https://ollama.com/install.sh | sh

# Comandos essenciais
ollama run llama3.2        # baixa e inicia chat interativo
ollama run mistral         # outro modelo
ollama list                # listar modelos instalados
ollama pull llama3.2       # baixar sem iniciar
ollama rm llama3.2         # remover modelo
ollama serve               # iniciar servidor na porta 11434

# API REST (Ollama roda um servidor HTTP local)
curl http://localhost:11434/api/generate -d '{
  "model": "llama3.2",
  "prompt": "Explique machine learning em 2 frases",
  "stream": false
}'

LM Studio — interface visual

LM Studio e um aplicativo desktop que oferece interface grafica para baixar, gerenciar e conversar com modelos locais. Ideal para quem prefere GUI ao terminal. Suporta modelos no formato GGUF (quantizados) do Hugging Face.

Quantizacao e o processo de reduzir a precisao dos pesos do modelo (de float32 para int4, por exemplo) para caber em menos RAM. Q4_K_M e o sweet spot para a maioria dos casos: excelente qualidade com uso de memoria significativamente menor. Q8_0 e mais fiel ao modelo original mas usa mais RAM.

O que e Claude Code

Claude Code e um agente de desenvolvimento AI que roda no seu terminal. Diferente de um chatbot, ele tem acesso direto ao seu sistema de arquivos, pode executar comandos, ler e escrever codigo, fazer buscas no repositorio e interagir com ferramentas externas via MCP. E projetado para trabalhar dentro do seu projeto, com todo o contexto do codebase disponivel.

A principal vantagem sobre IDEs com AI embutida (Cursor, Copilot) e o controle. Claude Code e transparente sobre o que vai fazer, pede confirmacao antes de acoes irreversiveis e deixa voce no controle total do fluxo. E uma ferramenta para desenvolvedores que querem entender o que esta acontecendo, nao so apertar "aceitar tudo".

O que e CLAUDE.md e por que e essencial

CLAUDE.md e um arquivo Markdown na raiz do seu projeto que e automaticamente carregado no contexto de cada sessao do Claude Code. E o equivalente a um "briefing permanente" — voce escreve uma vez, e o agente le toda vez que abrir o projeto. Sem ele, voce precisa reexplicar o stack, convencoes e regras toda sessao.

E a diferenca entre um assistente que escreve var quando seu projeto usa const, e um que segue exatamente suas convencoes de codigo. Ou entre um agente que comita direto na main e um que sempre abre PR.

O Fluxo Explore → Plan → Implement → Commit

O maior erro de quem comeca com Claude Code e pedir implementacao imediata em tarefas complexas. O fluxo correto tem quatro etapas que reduzem drasticamente retrabalho e alucinacoes:

  • ▶ Explore: "Leia os arquivos X, Y, Z e me explique como o sistema atual funciona." Claude Code le o codigo e cria um mapa mental sem modificar nada.
  • ▶ Plan: "Com base no que voce leu, proponha um plano para implementar [feature]. Nao comece ainda." Voce revisa e ajusta o plano.
  • ▶ Implement: "Ok, execute o plano." Agora Claude Code age com contexto completo.
  • ▶ Commit: Voce revisa, testa e commita. Nunca deixe Claude commitar sem revisao humana.

Plan Mode (Ctrl+G)

Plan Mode e um modo especial onde Claude Code pode explorar o codebase — ler arquivos, fazer buscas, analisar a estrutura — mas fica proibido de fazer qualquer modificacao. E a ferramenta perfeita para a fase de Explore sem risco de efeitos colaterais.

Use Ctrl+G para alternar entre Plan Mode e modo normal. No Plan Mode, uma indicacao visual aparece no terminal. Claude Code vai ler, analisar e planejar, mas qualquer tentativa de escrever arquivos sera bloqueada automaticamente.

Gerenciando Contexto — /clear e /compact

Contexto e o recurso mais precioso em uma sessao. Quanto mais longa a conversa, mais tokens sao consumidos e mais o modelo pode "se distrair" com informacoes antigas irrelevantes. Duas estrategias para gerenciar:

  • ▶ /clear: Limpa todo o contexto. Use entre tarefas completamente diferentes. Contexto de "corrigir um bug no checkout" nao ajuda em "criar uma pagina de perfil".
  • ▶ /compact com instrucao: Resume mantendo o essencial. Use quando a sessao ficou longa mas voce quer continuar. Exemplo: /compact Keep: decisions made, files modified, next steps.

O que e MCP e por que e importante

Model Context Protocol (MCP) e um padrao aberto criado pela Anthropic em novembro de 2024 e doado para a Linux Foundation — garantindo neutralidade e adocao pela industria. Pense nele como o "USB-C para AI": um conector universal que permite que qualquer LLM se integre com qualquer ferramenta ou fonte de dados usando o mesmo protocolo.

Antes do MCP, cada ferramenta AI tinha integracao proprietaria com cada servico. Com MCP, qualquer servidor MCP funciona com qualquer cliente MCP — Claude Code, Claude Desktop, Cursor, Cline, e qualquer outro cliente que implemente o protocolo. E por isso todos os grandes players (OpenAI, Google, Microsoft) adotaram rapidamente.

Arquitetura Cliente — Servidor

MCP usa uma arquitetura cliente-servidor simples. O cliente (Claude Code, Claude Desktop) gerencia a conexao com os servidores e inclui as tools disponibilizadas no contexto do LLM. O servidor e um processo que expoe tools — pode ser um script Node.js local, um container Docker, ou um servico remoto.

Quando voce instala um MCP server, o cliente descobre automaticamente as tools que ele oferece. Claude Code pode entao usar essas tools como usa Read, Edit e Bash — mas agora com acesso a sistemas externos como GitHub, Supabase, Figma ou qualquer outro servico que tenha um MCP server.

Instalando MCP Servers no Claude Code

# Listar servers instalados
claude mcp list

# Adicionar um server
claude mcp add filesystem -- --allowed-directory /Users/voce/projetos

# Remover um server
claude mcp remove filesystem

# Ver detalhes de um server
claude mcp get filesystem

# Servidores ficam em ~/.config/claude/mcp_servers.json

MCP Tool Search — Lazy Loading

Um problema com muitos MCP servers instalados e que todas as tools ficam carregadas no contexto — consumindo tokens mesmo quando nao sao usadas. O MCP Tool Search resolve isso com lazy loading: as tools so sao carregadas quando necessarias, reduzindo o uso de contexto em ate 95%.

O limite pratico para MCP servers simultaneos e 5-6. Acima disso, o contexto fica saturado e a qualidade das respostas cai. Use Tool Search para projetos com muitas integracoes.

Agentes e Multi-Agentes

Construindo sistemas que trabalham juntos de forma autonoma

O que e um Agente AI

Um agente AI e um sistema que usa um LLM como nucleo de decisao, mas vai alem do simples question-answer. Ele percebe o ambiente, decide acoes, executa essas acoes com ferramentas, e observa o resultado para planejar o proximo passo — iterando ate atingir o objetivo ou decidir que e impossivel.

A definicao pratica: um agente e qualquer sistema onde o LLM controla o loop de execucao. Diferente de um chatbot (responde e para) ou copilot (sugere, humano executa), o agente age autonomamente por multiplos passos.

Os 4 Componentes Essenciais

  • ▶ LLM (Cerebro): O modelo de linguagem que raciocina, planeja e decide. Claude, GPT-4o, Gemini — a escolha afeta qualidade e custo do sistema inteiro.
  • ▶ Tools (Maos): As capacidades de acao do agente. Sem tools, e apenas um chatbot. Com tools, pode ler arquivos, chamar APIs, executar codigo, buscar na web.
  • ▶ Memory (Memoria): Como o agente persiste informacao. Pode ser in-context (dentro da janela de tokens), externo (banco vetorial, arquivo), ou hibrido.
  • ▶ Goal (Objetivo): O que o agente esta tentando alcancar. Pode ser definido por prompt, por CLAUDE.md, ou por um orquestrador que delega sub-objetivos.

O Agent Loop

Todo agente opera em um loop continuo ate atingir o objetivo ou ser interrompido:

1. Observe → 2. Plan → 3. Act → 4. Reflect → repeat ou Done 
# Agente simples em Python
import anthropic
import json

client = anthropic.Anthropic()

# Definicao de tools
tools = [
    {
        "name": "read_file",
        "description": "Le o conteudo de um arquivo",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string", "description": "Caminho do arquivo"}
            },
            "required": ["path"]
        }
    },
    {
        "name": "write_file",
        "description": "Escreve conteudo em um arquivo",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string"},
                "content": {"type": "string"}
            },
            "required": ["path", "content"]
        }
    }
]

def execute_tool(name, inputs):
    if name == "read_file":
        with open(inputs["path"]) as f:
            return f.read()
    elif name == "write_file":
        with open(inputs["path"], "w") as f:
            f.write(inputs["content"])
        return f"Arquivo {inputs['path']} escrito com sucesso"

# Agent loop
messages = [{"role": "user", "content": "Leia o arquivo README.md e crie um resumo em SUMMARY.md"}]

while True:
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=4096,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "end_turn":
        print("Agente concluiu a tarefa")
        break

    # Processar tool calls
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result
            })

    # Continuar o loop
    messages.append({"role": "assistant", "content": response.content})
    messages.append({"role": "user", "content": tool_results})

O orquestrador e o agente central que recebe o objetivo de alto nivel e decide como dividi-lo em sub-tarefas para agentes especializados. A regra de ouro: o orquestrador nunca implementa. Ele planeja, delega, verifica resultados e agrega. Qualquer implementacao direta e sinal de design errado.

Por que Regras Compartilhadas sao Essenciais

Sem regras compartilhadas, cada agente desenvolve seu proprio estilo e comportamento. O agente de copywriting escreve em tom formal enquanto o de email usa tom casual. O agente de codigo usa var, outro usa const. Em sistemas multi-agentes, consistencia requer enforcement sistematico, nao esperanca.

Git Worktrees — A Base do Paralelismo

Git worktrees sao copias "leves" do repositorio — compartilham o historico git mas tem o working directory separado. Cada worktree pode estar em uma branch diferente, o que permite que multiplos agentes trabalhem em paralelo sem conflito. Um agente pode estar implementando a feature A enquanto outro testa a feature B.

A vantagem sobre clonar o repo multiplas vezes e que worktrees compartilham o .git — ocupam muito menos espaco em disco e mudancas em uma worktree ficam visiveis para outras instantaneamente.

# Criar worktrees para agentes paralelos
git worktree add ../projeto-agent-1 -b feature/agent-1
git worktree add ../projeto-agent-2 -b feature/agent-2
git worktree add ../projeto-agent-3 -b feature/agent-3

# Listar worktrees
git worktree list

# Em cada terminal separado:
cd ../projeto-agent-1 && claude  # Agente 1 trabalhando
cd ../projeto-agent-2 && claude  # Agente 2 em paralelo
cd ../projeto-agent-3 && claude  # Agente 3 em paralelo

# Remover worktree quando terminar
git worktree remove ../projeto-agent-1

O Padrao Writer/Reviewer

Um dos patterns mais eficientes para qualidade: um agente escreve e outro revisa — em paralelo, usando worktrees. O writer implementa a feature. O reviewer, em outra sessao com a mesma codebase, analisa o codigo sem ter sido influenciado pelo processo de escrita. Resultados: reviews mais criticos, mais bugs capturados.

O Conceito do CrewAI

CrewAI modela agentes como membros de uma equipe (crew) com papeis bem definidos. Cada agente tem um role (cargo), goal (objetivo pessoal), backstory (contexto que define personalidade e expertise) e tools (capacidades). Tasks sao atribuidas a agentes com expected_output definido.

A metafora da equipe e poderosa porque alinha com como humanos pensam sobre divisao de trabalho. O processo pode ser Sequential (uma tarefa por vez, output vira input da proxima) ou Hierarchical (um manager agent decide quem faz o que).

# pip install crewai crewai-tools
from crewai import Agent, Task, Crew, Process
from crewai_tools import SerperDevTool

search_tool = SerperDevTool()

# Definindo a equipe
researcher = Agent(
    role="Pesquisador de Mercado Senior",
    goal="Descobrir tendencias emergentes em AI para negocios B2B",
    backstory="""Voce e um analista com 10 anos de experiencia em tech B2B.
    Especializado em identificar oportunidades antes do mainstream.
    Metodico e data-driven.""",
    tools=[search_tool],
    verbose=True,
    llm="claude-3-5-sonnet-20241022"
)

writer = Agent(
    role="Redator de Conteudo Tecnico",
    goal="Transformar pesquisa em artigos claros e acionaveis",
    backstory="""Escritor tecnico com background em engenharia.
    Traduz complexidade em linguagem executiva sem perder substancia.
    Tom: consultivo, direto, sem hype.""",
    verbose=True,
    llm="claude-3-5-sonnet-20241022"
)

reviewer = Agent(
    role="Editor Senior",
    goal="Garantir qualidade, precisao e impacto do conteudo",
    backstory="Editor com padrao jornalistico rigoroso. Elimina jargao, exige evidencias.",
    verbose=True,
    llm="claude-3-5-haiku-20241022"  # menor custo para revisao
)

# Definindo as tarefas
research_task = Task(
    description="Pesquise as 5 principais tendencias de AI para negocios B2B em 2025",
    expected_output="Relatorio com 5 tendencias, evidencias de mercado e implicacoes praticas",
    agent=researcher
)

writing_task = Task(
    description="Baseado na pesquisa, escreva um artigo de 800 palavras para executivos",
    expected_output="Artigo completo em portugues, pronto para publicacao",
    agent=writer,
    context=[research_task]  # depende da tarefa de pesquisa
)

review_task = Task(
    description="Revise o artigo: precisao tecnica, clareza, tom e impacto",
    expected_output="Artigo revisado com melhorias implementadas e justificadas",
    agent=reviewer,
    context=[writing_task]
)

# Montando e rodando o crew
crew = Crew(
    agents=[researcher, writer, reviewer],
    tasks=[research_task, writing_task, review_task],
    process=Process.sequential,
    verbose=True
)

result = crew.kickoff()
print(result)

A Abordagem de Grafos

LangGraph representa workflows como grafos dirigidos: nos sao agentes ou funcoes de processamento, arestas definem as transicoes entre nos — podendo ser condicionais (se X faz Y, senao faz Z). O estado e compartilhado entre todos os nos e persiste ao longo do grafo.

A vantagem sobre frameworks lineares e o controle preciso do fluxo. Voce pode implementar loops (retornar a um no anterior), branches condicionais complexas, e paradas humanas (human-in-the-loop) em qualquer ponto. Ideal para workflows com compliance, auditoria ou aprovacao humana necessaria.

# pip install langgraph langchain-anthropic
from langgraph.graph import StateGraph, END
from langchain_anthropic import ChatAnthropic
from typing import TypedDict, Literal

llm = ChatAnthropic(model="claude-3-5-sonnet-20241022")

# Estado compartilhado entre todos os nos
class WorkflowState(TypedDict):
    query: str
    research: str
    draft: str
    approved: bool
    feedback: str
    iterations: int

# Nos do grafo (cada um e uma funcao)
def research_node(state: WorkflowState) -> WorkflowState:
    response = llm.invoke(f"Pesquise: {state['query']}")
    return {**state, "research": response.content}

def writing_node(state: WorkflowState) -> WorkflowState:
    prompt = f"Com base na pesquisa:\n{state['research']}\nEscreva um artigo."
    response = llm.invoke(prompt)
    return {**state, "draft": response.content, "iterations": state["iterations"] + 1}

def review_node(state: WorkflowState) -> WorkflowState:
    prompt = f"Revise este artigo e responda APPROVED ou REVISE:\n{state['draft']}"
    response = llm.invoke(prompt)
    approved = "APPROVED" in response.content.upper()
    return {**state, "approved": approved, "feedback": response.content}

# Aresta condicional
def should_continue(state: WorkflowState) -> Literal["writing", END]:
    if state["approved"] or state["iterations"] >= 3:
        return END
    return "writing"  # volta para reescrever

# Construindo o grafo
workflow = StateGraph(WorkflowState)
workflow.add_node("research", research_node)
workflow.add_node("writing", writing_node)
workflow.add_node("review", review_node)

workflow.set_entry_point("research")
workflow.add_edge("research", "writing")
workflow.add_edge("writing", "review")
workflow.add_conditional_edges("review", should_continue)

app = workflow.compile()

# Executar
result = app.invoke({
    "query": "Tendencias AI para SaaS em 2025",
    "research": "", "draft": "", "approved": False,
    "feedback": "", "iterations": 0
})

LangGraph Studio

LangGraph Studio e uma interface visual para debugar grafos em tempo real. Voce ve o estado a cada no, pode pausar e injetar dados manualmente, e visualizar o fluxo de execucao. Essencial para debugar workflows complexos.

A Abordagem Conversacional

AutoGen (rebatizado AG2 pela comunidade) tem uma premissa diferente: agentes se comunicam em linguagem natural, como humanos em uma reuniao. Um agente pode questionar o outro, pedir esclarecimentos, propor alternativas e debater abordagens antes de chegar a uma decisao. Isso torna o processo mais robusto para problemas ambiguos.

O pattern mais classico e o AssistantAgent + UserProxyAgent: o assistant resolve problemas, o proxy representa o humano (pode executar codigo, aprovar acoes, ou ser totalmente automatico). O proxy tambem pode ter um LLM que simula o usuario para testes automatizados.

# pip install pyautogen
import autogen

config_list = [{
    "model": "claude-3-5-sonnet-20241022",
    "api_key": "sua-chave",
    "api_type": "anthropic"
}]

llm_config = {"config_list": config_list}

# Agente que escreve e testa codigo
coder = autogen.AssistantAgent(
    name="Desenvolvedor",
    llm_config=llm_config,
    system_message="""Voce e um desenvolvedor Python senior.
    Escreva codigo funcional, testado e com boas praticas.
    Apos escrever o codigo, revise possíveis bugs antes de enviar."""
)

# Agente que revisa e executa codigo
reviewer = autogen.AssistantAgent(
    name="Revisor",
    llm_config=llm_config,
    system_message="""Voce e um revisor critico de codigo.
    Identifique bugs, problemas de seguranca e sugestoes de melhoria.
    Seja especifico: cite linha e problema exato."""
)

# Proxy do usuario (pode ser humano ou automatico)
user_proxy = autogen.UserProxyAgent(
    name="Usuario",
    human_input_mode="NEVER",  # NEVER = totalmente automatico
    max_turns=10,  # CRITICO: sempre setar para evitar loops infinitos
    code_execution_config={
        "work_dir": "codigo_gerado",
        "use_docker": False
    }
)

# Iniciar conversa entre os 3 agentes
groupchat = autogen.GroupChat(
    agents=[user_proxy, coder, reviewer],
    messages=[],
    max_round=8
)
manager = autogen.GroupChatManager(groupchat=groupchat, llm_config=llm_config)

user_proxy.initiate_chat(
    manager,
    message="Crie uma funcao Python que valida CPF e inclua testes unitarios."
)

Importante — Loops Infinitos

Sempre configure max_turns e max_round. Sem esses limites, agentes podem entrar em loops custosos gerando centenas de dolares em tokens. Valores seguros: max_turns=10, max_round=8 para a maioria dos casos.

O Conceito de Handoff

O OpenAI Agents SDK (evolucao do Swarm experimental) introduziu o conceito de handoff: passagem de controle explicita de um agente para outro. Quando o agente A decide que a tarefa pertence ao agente B, ele faz o handoff — B recebe toda a conversa acumulada ate aquele ponto e continua.

O pattern mais util e o Triage Agent: um agente inicial que classifica a intencao do usuario e redireciona para o especialista correto. Simples, eficiente e facil de escalar com novos especialistas.

# pip install openai-agents
from agents import Agent, Runner, handoff

# Agentes especialistas
suporte_tecnico = Agent(
    name="Suporte Tecnico",
    instructions="""Voce resolve problemas tecnicos do produto.
    Foco em: bugs, erros, configuracao, integracao.
    Se o problema for sobre faturamento, faca handoff para financeiro.""",
    model="gpt-4o"
)

financeiro = Agent(
    name="Financeiro",
    instructions="""Voce resolve questoes de pagamento e faturamento.
    Foco em: cobrancas, cancelamento, upgrade de plano.
    Se o problema for tecnico, faca handoff para suporte tecnico.""",
    model="gpt-4o"
)

# Agente de triagem
triage = Agent(
    name="Triagem",
    instructions="""Voce classifica a solicitacao do usuario e direciona
    para o agente correto. Nao tente resolver — apenas classifique e redirecione.
    Use handoff para suporte_tecnico ou financeiro conforme o contexto.""",
    model="gpt-4o-mini",  # modelo menor para triagem simples
    handoffs=[
        handoff(suporte_tecnico, tool_name_override="escalar_para_tecnico"),
        handoff(financeiro, tool_name_override="escalar_para_financeiro")
    ]
)

# Runner com guardrails
async def main():
    result = await Runner.run(
        triage,
        input="Meu cartao foi cobrado duas vezes esse mes",
        max_turns=10
    )
    print(result.final_output)

import asyncio

Framework Decision Guide

A escolha do framework certo pode significar semanas de desenvolvimento vs dias. Use estas diretrizes para decidir rapidamente:

  • ▶ Use CrewAI quando: voce pensa em termos de papeis (researcher, writer, analyst). O problema mapeia bem para uma equipe humana. Quer comecar rapido com minimo de codigo. Multi-modelo.
  • ▶ Use LangGraph quando: o workflow tem condicoes complexas, loops e branches. Precisa de auditoria/compliance (cada passo fica registrado). Human-in-the-loop em pontos especificos. Debugar visualmente.
  • ▶ Use AutoGen quando: o valor esta na conversa entre agentes (debater, questionar, refinar). Execucao de codigo e central ao workflow. Quer simular um time de devs discutindo um problema.
  • ▶ Use OpenAI Agents SDK quando: prototipagem rapida com OpenAI. Handoff pattern e o core. Nao precisa de multi-modelo. Minimalismo de codigo e prioridade.
  • ▶ Use Claude Code Agents quando: o trabalho e desenvolvimento de software. Quer integracao nativa com o filesystem e tools de dev. Prefere arquivos Markdown a codigo Python para definir agentes.
